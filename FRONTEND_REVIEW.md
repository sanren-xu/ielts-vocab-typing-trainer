# FRONTEND_REVIEW.md

## 1. 文件用途

本文件用于记录 IELTS Typing Learner 的前端页面复盘和商业化版本页面改造计划。

当前用户已经明确反馈：

对现在的页面已有功能不满意，商业化版本不能直接把旧页面硬接后端。

因此，在继续写后端、数据库和支付之前，需要先复盘前端页面。

本文件主要回答：

1. 当前有哪些页面
2. 每个页面现在负责什么
3. 当前页面有哪些问题
4. 商业化版本需要哪些页面
5. 哪些页面保留
6. 哪些页面重做
7. 哪些页面新增
8. 哪些页面先做静态原型
9. 哪些页面后续接 API
10. 哪些竞品功能值得参考
11. 哪些想法暂时只记录，不立刻开发

## 3. 当前主流程

当前本地版主流程是：

1. 打开首页
2. 创建或进入词库
3. 录入单词
4. 管理词库
5. 进入普通练习
6. 答错后生成错词
7. 进入错词页
8. 练习错词
9. 在设置页导入 / 导出 / 清空数据

这个流程已经能跑通，但还不够适合商业化上线。

当前主要问题不是“功能完全不能用”，而是：

1. 页面目标还不够清楚
2. 新用户引导不足
3. 学习进度反馈不够直观
4. 练习页设置不够集中
5. 错词页还可以做成差异点
6. 商业化版本需要登录、会员、支付、反馈、后台等新页面
7. localStorage 页面不能直接硬接 MySQL 和后端 API

## 4. 商业化版本前端总原则

商业化版本不能简单地把当前页面接上后端。

原因：

1. 当前页面主要为本地 localStorage 设计
2. 商业化版本需要注册 / 登录
3. 商业化版本需要会员状态
4. 商业化版本需要免费额度
5. 商业化版本需要订单和支付入口
6. 商业化版本需要用户反馈
7. 商业化版本需要管理后台
8. 用户第一次访问时需要更清楚的引导
9. 统计、练习设置、错词页需要重新规划

商业化前端改造原则：

1. 先复盘页面，不急着接 API
2. 先确定页面结构，再写后端接口
3. 页面数据来源要明确：localStorage、API，还是临时 mock data
4. 不盲目引入 React / Vue
5. 优先复用现有 `style.css`
6. 不要重写整个项目
7. 每次只改一个页面或一个小区域
8. 每次改完都要说明测试方式
9. 首页保持清爽，不直接堆复杂图表
10. 统计图表单独放到 `stats.html`
11. 错词页可以作为产品差异点重点打磨

## 5. index.html 页面复盘

### 当前定位

`index.html` 当前是首页入口。

主要作用：

1. 进入录入页
2. 进入词库页
3. 进入练习页
4. 进入错词页
5. 进入设置页

### 当前可能存在的问题

1. 更像功能入口，不像学习首页
2. 缺少学习进度反馈
3. 缺少当前词库状态
4. 缺少今日新增词数量
5. 缺少当前错词数量
6. 缺少最近练习时间
7. 对新用户来说，不一定知道第一步该做什么
8. 商业化版本缺少登录状态和会员状态展示

### 商业化版本首页原则

首页不要直接放复杂统计图表。

首页应该放：

1. 主要功能入口
2. 当前学习状态的简短提示
3. 明显的“学习统计”入口
4. 明显的“开始练习”入口
5. 明显的“录入单词”入口
6. 当前登录状态
7. 当前会员状态提示

首页不直接展示复杂图表。

图表和详细统计应该放到单独页面：

`stats.html`

### 首页建议新增入口

首页可以新增一个明显入口：

1. 学习统计
2. 查看学习进度
3. 数据统计
4. 我的学习趋势

入口形式可以是：

1. 一个按钮
2. 一个卡片
3. 一个轻量统计区里的“查看详情”

### 首页可以显示的轻量信息

首页可以显示少量核心数字，但不放完整图表。

例如：

1. 当前词库数
2. 当前单词数
3. 当前错词数
4. 最近练习时间
5. 今日新增词数

这些数字只作为概览。

完整图表进入 `stats.html` 查看。

## 6. stats.html 页面规划

### 页面定位

`stats.html` 是学习统计页面。

它专门用来展示学习数据和图表，不让首页变复杂。

### 为什么单独做统计页

原因：

1. 首页需要保持清爽
2. 图表和数据较多，放首页会干扰主流程
3. 统计页可以集中展示学习进度
4. 后续商业化版本可以逐步增强统计能力
5. 设置页和首页都可以跳转到统计页

### stats.html 的入口

入口可以放在：

1. `index.html` 首页
2. `settings.html` 设置页
3. 未来用户菜单中

优先级：

1. 首页放主入口
2. 设置页放辅助入口

### V1 统计页先做简单版本

V1 统计页可以先做：

1. 总词库数
2. 总单词数
3. 当前错词数
4. 累计练习轮次
5. 最近练习时间
6. 今日新增词数
7. 近 7 天练习次数
8. 一个简单趋势图或柱状图
9. 按词库统计单词数和错词数

### 图表原则

图表不用一开始做得复杂。

先做一个明显、直观的图即可。

可选图表：

1. 近 7 天练习轮次柱状图
2. 近 7 天新增单词柱状图
3. 各词库单词数量柱状图
4. 错词数量变化趋势图

第一版建议优先：

近 7 天练习轮次柱状图。

原因：

1. 容易理解
2. 数据容易计算
3. 能直观看到学习是否持续
4. 不需要复杂算法

### 数据来源

本地版：

1. 从 localStorage 读取
2. 根据 library 和 word 字段计算
3. 根据 lastPracticedAt 和 totalPracticeRounds 计算

commercial-v1：

1. 从 MySQL 读取
2. 通过后端 API 返回统计数据
3. 不让前端直接查数据库

### 后续可能需要的 API

后续 commercial-v1 可能需要：

1. `GET /api/stats/overview`
2. `GET /api/stats/recent-practice`
3. `GET /api/stats/libraries`
4. `GET /api/stats/wrong-words`

但现在不用急着设计 API，先完成页面定位。

1. 云端词库列表
2. 词库数量限制
3. 单词数量限制

5. 会员升级提示

但商业化版本需要重点优化：

1. 空状态提示
2. 新手引导
3. 当前词库状态
4. 免费额度提示
5. 练习入口
6. 错词入口

## 8. input.html 页面复盘

### 当前定位

`input.html` 是单词录入页。

当前功能：

1. 输入英文单词
2. 输入中文意思
3. 输入例句
4. 新建今日词库
5. 追加到当前词库
6. 保存单词

### 当前可能存在的问题

1. 用户可能不清楚当前保存到哪个词库
2. 新建今日词库的逻辑需要更清楚
3. 保存成功反馈可以更明显
4. 重复单词提醒可以更友好
5. 录入后下一步动作不够明确
6. 例句是否必填需要更清楚
7. 商业化版本要考虑免费额度限制

### 商业化版本可能需要承担的功能

1. 云端保存单词
2. 免费用户录入数量限制
3. 当前词库选择
4. 重复单词提醒
5. 保存并继续
6. 保存后去练习
7. 保存后返回词库
8. 保存失败提示
9. 登录过期提示
10. 会员升级提示

### 关于扩展词条信息

从 Type Words / 词文记得到的启发：

未来可以考虑给单词增加更多信息，例如：

1. 短语
2. 同义词
3. 同根词
4. 词源
5. 备注
6. 语境来源

但这些不建议 V1 一开始全部做。

原因：

1. 会让录入页变复杂
2. 会增加数据库字段
3. 会增加练习页提示逻辑
4. 会让新用户不知所措
5. 会拖慢 V1 上线

V1 录入页仍然优先保留最核心字段：

1. 英文
2. 中文
3. 例句
4. 所属词库

未来可以用“高级信息”折叠区处理扩展字段。

### 建议方向

`input.html` 是核心页面，应优先优化。

优先改进：

1. 明确当前词库
2. 增强保存成功反馈
3. 增加保存后操作
4. 避免空词库问题
5. 优化重复单词提示
6. 预留免费额度提示区域
7. 未来再考虑高级词条信息

## 9. practice.html 页面复盘

### 当前定位

`practice.html` 是拼写练习页。

当前支持：

1. 普通练习
2. 错词模式练习
3. 中文提示
4. 例句提示
5. 输入英文
6. 判断正确 / 错误
7. 记录错词
8. 一轮结束统计
9. 更新 V2 练习统计字段

### 当前可能存在的问题

1. 练习开始前的规则说明可能不足
2. 一轮练习结束后的成就感可以加强
3. 错词和正确词的反馈可以更清楚
4. 练习统计可以更直观
5. 未来需要云端保存练习轮次
6. 未来需要处理登录状态
7. 未来需要处理免费练习次数或单词数限制
8. 练习设置还不够集中

### 商业化版本可能需要承担的功能

1. 从后端读取练习词
2. 提交练习结果到后端
3. 更新单词统计
4. 更新错词状态
5. 更新用户练习记录
6. 检查会员状态
7. 检查免费额度
8. 练习结束页
9. 会员提示
10. 练习历史入口

### 练习页设置

参考竞品中的练习设置，未来 `practice.html` 可以集中放一些轻量设置。

V1 可考虑：

1. 选择词库
2. 练习顺序：录入顺序 / 随机顺序
3. 例句显示方式
4. 是否显示上一个 / 下一个单词
5. 是否练习错词
6. 是否开启单词循环练习

暂时不建议一开始做太复杂：

1. 错一次显示什么
2. 错三次显示什么
3. 多级提示策略
4. 自动切换难度
5. 高级词根词源提示

这些可以记录为未来功能。

### 是否显示释义 / 答案

竞品中有开关可以显示释义或答案，使用上很方便。

但问题是：

1. 对不自律用户可能削弱练习效果
2. 容易变成看答案
3. 需要清楚区分“提示”和“答案”

本项目建议：

1. V1 不直接做“显示答案”开关
2. 可以做“显示提示”设置
3. 可以保留例句提示
4. 后续再讨论是否允许显示答案

### 单个单词循环

竞品中的单个单词循环功能值得记录。

本项目可以考虑未来加入：

1. 当前单词重复练习
2. 错词重点循环
3. 高频错词循环
4. 用户手动选择某个词循环

优先级：

V1 可选，不是必须。

### 即时统计

竞品练习过程中有即时统计。

当前项目是一轮结束后显示统计。

建议：

1. V1 先保留一轮结束统计
2. 不急着做复杂实时统计
3. 可以先在练习页显示轻量进度，例如第几个词 / 总共几个词
4. 完整统计进入 `stats.html`

### 建议方向

`practice.html` 是产品核心价值页面，不能大改乱改。

优先保持：

1. 手打输入
2. 中文提示
3. 例句可选
4. 错词记录
5. 错词练习

后续增强：

1. 练习设置
2. 选择词库
3. 练习结束统计
4. 成就反馈
5. 错词回顾
6. 云端保存
7. 免费额度判断

## 10. wrong.html 页面复盘

### 当前定位

`wrong.html` 是错词复习页。

当前功能：

1. 按词库查看错词
2. 显示有错词的词库
3. 查看错词列表
4. 进入错词练习
5. 支持错词统计
6. 支持错词库顺序

### 当前可能存在的问题

1. 错词页对新用户解释不够
2. 没有错词时的空状态可以更友好
3. 错词移出规则需要更明确
4. 错词严重程度可以更直观
5. 商业化版本需要云端错词状态

### 商业化版本可能需要承担的功能

1. 从后端读取错词
2. 显示每个词库错词数量
3. 显示错词次数
4. 显示连续正确次数
5. 进入错词练习
6. 云端更新错词状态
7. 会员用户错词复习权益

### 错词页应成为产品差异点

用户认为 Qwerty Learner 的错词本比较简陋，本项目可以在错词上多下功夫。

错词页可以重点展示：

1. 错词次数
2. 连续正确次数
3. 是否仍在错词本
4. 所属词库
5. 最近练习时间
6. 错词严重程度
7. 一键练习当前词库错词
8. 一键练习全部错词

### 错词严重程度标记

参考 Type Words / 词文记的思路，每个单词可以记录错的次数，不同次数给不同标记。

当前本地版已有类似字段：

1. `wrongRoundCount`
2. `correctStreak`
3. `inWrongBook`
4. `practiceCount`
5. `correctCount`

未来页面可以显示：

1. 错 1 次：轻度错词
2. 错 3 次：重点错词
3. 错 5 次及以上：高频错词

具体文案可以后续再定。

### 生图功能

用户提到未来希望错词相关可以有“生图功能”。

这个属于理想功能，暂时只记录。

可能方向：

1. 生成错词复习图
2. 生成学习进度图
3. 生成打卡图
4. 生成某个词库的错词总结图

当前不进入 V1。

原因：

1. 不是核心闭环必须功能
2. 可能涉及截图、画布或后端生成
3. 容易拖慢上线
4. 可以等产品有用户后再判断

### 建议方向

`wrong.html` 保留，但需要加强说明。

优先改进：

1. 空状态
2. 错词规则说明
3. 错词练习入口
4. 错词数量展示
5. 连续正确移出错词的提示
6. 错词严重程度标记

## 21. 前端和后端的配合原则

不能简单按照：

前端全部做完  
→ 再写后端  
→ 再设计数据库

更合理的顺序是：

1. 先复盘前端
2. 明确页面结构
3. 明确每个页面需要的数据
4. 设计 API
5. 设计数据库
6. 写后端
7. 前端先用 mock data 做静态原型
8. 再逐步接真实 API

页面开发前要先判断：

1. 这个页面是否需要登录
2. 这个页面是否需要会员状态
3. 这个页面是否读取云端数据
4. 这个页面是否写入云端数据
5. 这个页面是否可以先用 localStorage 或 mock data
6. 这个页面是否影响现有本地版主流程
