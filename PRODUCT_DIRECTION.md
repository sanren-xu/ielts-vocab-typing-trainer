# IELTS Typing Learner 产品定位与开发边界

## 一、产品一句话定位

面向雅思机考考生的自建生词库手打练习工具。

更短版本：

雅思机考生词手打练习器。

## 二、目标用户

第一版目标用户：

准备雅思机考、以前主要靠看词或手写背词，但对键盘输入英文单词不熟悉的学生。

潜在扩展用户：

1. 托福 / PTE / 多邻国等机考英语考试用户
2. 四六级、考研英语等需要英文拼写训练的人
3. 其他需要练习英文手打单词的人

第一版不要做太宽，优先服务雅思机考用户。

## 三、用户痛点

核心痛点：

手写单词会，机打会生。

很多考生以前背单词靠看、抄、手写，但雅思机考需要在电脑上快速准确输入英文。看得懂不代表打得准，手写熟不代表键盘熟。

## 四、核心使用场景

用户做完一篇雅思阅读或听力后，把自己不会拼、不熟悉、容易混的单词手动录入词库。

之后通过中文释义或例句提示，用键盘输入英文单词。输错的词进入错词复习，反复练到熟悉。

用户逐渐形成自己的雅思真题生词资产。

## 五、核心差异

本工具不是大而全背单词软件，也不是完整雅思模考平台，而是专注于：

1. 自建生词库
2. 键盘输入练习
3. 错词复习
4. 个人积累感
5. 雅思机考场景

## 六、和 Qwerty Learner 的区别

Qwerty Learner 更偏现成词库、键盘输入训练和泛英语学习。在公开网页体验中，它没有明显的普通用户直接创建个人词库入口。

本项目更强调：

1. 用户自己从雅思真题、阅读文章、听力材料中逐个录入生词
2. 每个人创建属于自己的单词库
3. 通过中文释义或例句提示手打英文
4. 错词自动沉淀
5. 词库逐渐变厚，形成积累感

## 七、和 Anki 的区别

Anki 很强，可以自建卡片，也支持 type in the answer 这类输入答案卡片。

但 Anki 对普通学生门槛较高，需要理解 deck、note、field、card type、template 等概念。

本项目要比 Anki 更简单：

打开网页 -> 创建词库 -> 录入单词 -> 开始手打练习 -> 错词复习。

## 八、和百词斩的区别

百词斩是成熟背单词平台，重点是内置词书、图像记忆、发音例句和移动端背词体验。

本项目不和百词斩比词库规模，也不做大而全背词平台。

本项目只解决一个更窄的问题：

用户如何把自己遇到的雅思生词变成能在电脑上打出来的个人词库。

## 九、和雅思模考平台的区别

雅思模考平台解决的是听说读写整套考试模拟。

本项目只解决一个小场景：

雅思备考中遇到的生词，如何通过键盘输入练熟。

## 十、不做什么

第一版明确不做：

1. 不做完整雅思模考
2. 不做听说读写全科训练
3. 不做 AI 写作批改
4. 不做口语陪练
5. 不做大规模内置词库平台
6. 暂时不做批量导入
7. 暂时不做账号系统和云同步
8. 暂时不做社交排名、社区、课程售卖
9. 暂时不做复杂算法推荐

## 十一、第一版成功标准

第一版成功不是赚钱，而是验证用户是否愿意真的用它积累单词。

成功标准：

1. 新用户 3 分钟内能理解基本流程
2. 用户能独立完成：创建词库 -> 录入 5 个词 -> 练习 -> 产生错词 -> 错词复习
3. 找 5 个真实用户试用，至少 3 个能独立完成流程
4. 至少 2 个用户第二天愿意再次打开
5. 用户能说清楚它和普通背单词软件有什么不同

## 十二、下一步开发方向

暂时不要继续随机加功能。

下一步优先级：

1. 继续做竞品调研，只看普通用户可直接访问的网站，不把 GitHub 仓库当直接竞品
2. 设计 `DATA_MODEL_V2.md`，整理单词、词库、练习记录、掌握状态、每日统计的数据结构
3. 后续再开发首页学习统计和积累感
4. 再找真实用户试用
5. 根据反馈决定是否做账号、云同步、付费功能

## 十三、开发原则

1. 每次只做一个小功能
2. 开始前先 `git status`
3. 如果不是 clean，先说明未提交文件，不要直接乱改
4. 不要改 `app.html` / `bulk.html`
5. 不要引入 React、Vue、Node、数据库或后端
6. 不要重写项目
7. 样式复用 `style.css`
8. 做完后说明修改文件和测试方式
