# PRODUCT_DIRECTION.md

## 1. 产品定位

IELTS Typing Learner 是一个面向雅思机考考生的个人生词手打练习工具。

核心不是“大词库”，也不是“完整雅思模考”，而是：

用户把自己在雅思阅读、听力、写作、课程、文章和复习资料中遇到的生词，手动录入个人词库，然后通过键盘输入练习英文拼写，并通过错词复习把词练熟。

一句话定位：

面向雅思机考考生的个人生词库手打练习工具。

---

## 2. 为什么要做这个产品

雅思机考环境下，考生不仅要认识单词，还要能快速、准确地输入英文。

很多背单词工具的问题是：

1. 词库太大，和用户真实学习材料脱节
2. 用户只是“认识”单词，但没有真正练拼写
3. 单词来源不是用户自己的学习过程
4. 练习过程缺少键盘输入
5. 错词复习不够贴近用户自己的薄弱点

本项目的核心价值是：

把用户自己遇到的雅思生词，转化成可反复输入练习的个人词库。

---

## 3. 目标用户

主要用户：

1. 准备雅思机考的学生
2. 雅思阅读中经常遇到生词的人
3. 雅思听力中需要积累拼写的人
4. 雅思写作中想积累表达的人
5. 不想背大词库，只想练自己生词的人
6. 想通过键盘输入提高英文拼写熟练度的人

当前最优先服务的用户：

正在备考雅思、英语水平处于 CET-4 到 IELTS 过渡阶段、愿意自己积累生词并反复练习的学生。

---

## 4. 产品核心流程

当前核心流程：

1. 创建词库
2. 手动录入单词
3. 查看和管理词库
4. 进入普通拼写练习
5. 答错后自动进入错词
6. 在错词页查看错词
7. 进行错词练习
8. 通过连续答对逐步移出错词
9. 查看学习统计

这个流程应该保持简单，不要被复杂功能打断。

## 5. 当前双版本路线

项目现在有两条路线：

### main：本地稳定版

main 是当前稳定的本地版。

特点：

1. 纯前端
2. HTML / CSS / JavaScript
3. localStorage 保存数据
4. 不需要后端
5. 不需要数据库
6. 可以直接通过浏览器使用
7. 适合作为原型和稳定备份

main 分支的目标：

保留一个能独立运行的本地版，不被商业化改造破坏。

---

### commercial-v1：国内商业化 Web 版

commercial-v1 是商业化 Web 版开发分支。

允许引入：

1. Node.js
2. Express
3. MySQL
4. 后端 API
5. 注册 / 登录
6. 云端词库
7. 云端练习记录
8. 会员系统
9. 订单系统
10. 支付宝支付
11. 留言反馈
12. 简单管理后台
13. 部署和备案流程

commercial-v1 的目标：

把本地生词练习工具升级为可以注册、云端保存、付费使用的国内商业化 Web 产品。

---

## 6. commercial-v1 的 V1 功能边界

V1 要做：

1. 注册 / 登录
2. 云端保存词库
3. 云端保存单词
4. 云端保存练习记录
5. 错词复习
6. 首页学习统计
7. 免费额度限制
8. 一个会员套餐
9. 支付宝支付
10. 支付成功后自动开通会员
11. 留言反馈
12. 简单管理后台

V1 的核心闭环：

用户注册  
→ 创建词库  
→ 录入生词  
→ 练习  
→ 产生错词  
→ 错词复习  
→ 达到免费额度限制  
→ 购买会员  
→ 支付成功自动开通  
→ 继续使用

## 8. 和普通背单词软件的区别

普通背单词软件通常强调：

1. 大词库
2. 记忆曲线
3. 选择题
4. 词根词缀
5. 每日任务
6. 排行榜
7. 打卡

本项目强调：

1. 用户自己的生词
2. 手动录入
3. 键盘输入
4. 拼写熟练
5. 雅思机考场景
6. 错词复习
7. 小而实用

本项目不追求覆盖所有英语学习场景。

---

## 9. 和 Qwerty Learner 的区别

Qwerty Learner 的重点是：

1. 打字练习
2. 现成词库
3. 快速开始
4. 开源社区
5. 多种词库选择

本项目的重点是：

1. 雅思机考考生
2. 用户自己的生词库
3. 录入、管理、练习、错词复习一体化
4. 从个人学习材料中积累
5. 未来支持云端保存和会员服务

本项目不应该简单复制 Qwerty Learner。

---

## 10. 和雅思模考平台的区别

雅思模考平台通常做：

1. 听力模考
2. 阅读模考
3. 写作评分
4. 口语练习
5. 分数预测
6. 真题套题
7. 复杂报告

本项目不做完整模考。

本项目只解决一个更小的问题：

用户遇到的生词，如何通过键盘输入练熟。

---

## 11. 手动录入为什么重要

本项目不以批量导入为核心。

手动录入的价值：

1. 用户主动筛选自己真正不会的词
2. 每个词都有学习场景
3. 录入本身也是第一次复习
4. 用户对个人词库有积累感
5. 避免产品变成又一个大词库工具

批量导入可以未来考虑，但不能破坏核心定位。

当前原则：

先做好手动录入和个人词库体验，再考虑导入能力。

## 13. app.html / bulk.html 的处理原则

### app.html

app.html 属于旧页面 / 备份页面。

默认不修改。

如果商业化版本需要处理它，应先判断：

1. 保留
2. 废弃
3. 重定向
4. 合并到新入口
5. 删除前备份

不能在普通任务中顺手修改。

### bulk.html

bulk.html 属于旧批量导入页面 / 暂停维护页面。

默认不修改。

如果未来要恢复批量导入，必须先重新设计：

1. 导入格式
2. 数据校验
3. 重复单词处理
4. 错误提示
5. 是否影响手动录入的核心定位

## 15. 开发原则

1. 每次只做一个小功能
2. 开始前先看 git status 和 git branch
3. 工作区不 clean 不继续
4. 保护 main 分支
5. commercial-v1 允许商业化改造
6. 不重写项目
7. 不盲目引入框架
8. 不一次性做大而全功能
9. 不把敏感信息提交到 GitHub
10. 做完要说明修改文件和测试方式

核心原则：

保护 main，放开 commercial-v1；小步修改，重大变更先确认。
