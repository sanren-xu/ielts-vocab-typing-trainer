# CODEX_RULES.md

## 1. 文件定位

本文件是 IELTS Typing Learner 项目的 Codex 协作规则说明。

当前更重要的长期规则文件是：

- `AGENTS.md`

本文件保留作为人类可读的协作说明和历史规则整理。

如果本文件和 `AGENTS.md` 有冲突：

以 `AGENTS.md` 为准。

---

## 2. 项目当前状态

项目现在有两条路线：

1. `main`：本地稳定版
2. `commercial-v1`：国内商业化 Web 版 V1

两条路线的规则不同，不能混在一起。

---

## 3. main 分支规则

`main` 是本地稳定版。

特点：

1. 纯前端
2. HTML / CSS / JavaScript
3. localStorage 保存数据
4. 不需要后端
5. 不需要数据库
6. 不需要 Node.js 运行项目

`main` 分支默认不引入：

1. Node.js 后端
2. MySQL
3. 注册 / 登录
4. 支付系统
5. 会员系统
6. 服务器部署逻辑

`main` 的作用：

保留一个稳定可用的本地版。

---

## 4. commercial-v1 分支规则

`commercial-v1` 是国内商业化 Web 版开发分支。

允许引入：

1. Node.js
2. Express
3. MySQL
4. 后端 API
5. 注册 / 登录
6. 云端词库
7. 练习记录云端保存
8. 会员系统
9. 订单系统
10. 支付宝支付
11. 留言反馈
12. 简单管理后台
13. 部署和备案文档

`commercial-v1` 的目标：

跑通个人生词手打练习工具的商业化 Web 闭环。

---

## 5. 每次任务开始前必须检查

每次开始前，先执行：

1. `git status`
2. `git branch`

如果工作区不是 clean：

1. 停止
2. 说明未提交文件
3. 判断是否和当前任务有关
4. 等用户确认

不要在未确认的情况下继续修改。

---

## 6. 小步开发原则

每次只做一个小功能。

不要在一个任务里同时做多个系统。

推荐任务粒度：

1. 新增一个文档
2. 修改一个文档
3. 创建 backend 骨架
4. 新增一个接口
5. 修改一个页面的小问题
6. 修复一个 bug
7. 新增一个测试步骤

不推荐任务粒度：

1. 一次性写完整后端
2. 一次性重做所有页面
3. 一次性接入登录、会员、支付
4. 一次性重构所有 JS
5. 没有确认就改变技术栈

---

## 7. app.html / bulk.html 规则

### app.html

`app.html` 是旧页面 / 备份页面。

默认不修改。

只有在任务明确要求处理旧页面时，才可以修改。

修改前必须说明：

1. 为什么要改
2. 如何改
3. 是否影响当前主流程
4. 如何测试

### bulk.html

`bulk.html` 是旧批量导入页面 / 暂停维护页面。

默认不修改。

如果未来要恢复批量导入，必须先重新设计导入逻辑，不要随手修复或扩展。

---

## 8. 文档优先原则

现在项目处于：

第 2.5 阶段：文档重整与前端复盘

当前不建议立刻写后端。

当前优先级：

1. `AGENTS.md`
2. `DOC_INDEX.md`
3. `PRODUCT_DIRECTION.md`
4. `PROJECT_STATUS.md`
5. `README.md`
6. `FRONTEND_REVIEW.md`
7. `CODEX_RULES.md`

文档重整完成后，再进入：

1. 前端复盘
2. 后端骨架
3. 数据库设计
4. API 设计
5. 注册登录
6. 云端词库
7. 支付系统

---

## 9. 安全规则

不要提交敏感信息。

禁止提交：

1. `.env`
2. 数据库密码
3. MySQL root 密码
4. 支付宝私钥
5. 支付宝公钥
6. 支付证书
7. API Key
8. 真实用户数据
9. 生产数据库备份

如果需要环境变量，只能提交 `.env.example`。

---

## 10. 依赖规则

新增 npm 依赖前必须说明：

1. 依赖名称
2. 用途
3. 为什么需要
4. 是否有替代方案
5. 是 runtime dependency 还是 dev dependency

不要为了方便随便安装大量依赖。

---

## 11. 数据库规则

`commercial-v1` 使用 MySQL。

前端不能直接连接 MySQL。

正确结构：

前端页面  
→ 后端 API  
→ MySQL 数据库

修改数据库前必须说明：

1. 表名
2. 字段
3. 影响范围
4. 是否影响旧数据
5. 如何验证
6. 是否有回滚方式

禁止随便执行：

1. `DROP TABLE`
2. `TRUNCATE TABLE`
3. 大范围 `DELETE`

除非用户明确确认这是本地测试。

---

## 12. API 规则

新增 API 时，必须说明：

1. 请求方法
2. 请求路径
3. 请求参数
4. 返回格式
5. 错误情况
6. 是否需要登录
7. 如何用 Apifox 测试

不要只写接口，不给测试方式。

---

## 13. 支付规则

支付系统不要提前做。

接支付宝前必须先完成：

1. 用户系统
2. 会员状态
3. 订单系统
4. 支付流程文档
5. 支付回调设计
6. 验签逻辑设计

不能通过前端判断支付成功来开通会员。

会员开通必须以后端收到支付宝异步通知并验签成功为准。

---

## 14. 提交规则

不要自动提交。

用户明确要求 commit 时，先执行：

1. `git status`
2. `git diff --stat`

然后只添加本次相关文件。

不要随便使用：

`git add .`

除非用户明确确认。

---
